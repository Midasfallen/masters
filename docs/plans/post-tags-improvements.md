# План доработок: отметки пользователей и хештеги в постах

## Context

После ручного тестирования фичи «отметки пользователей в посте» пользователь выявил
5 проблем. Расследование показало, что часть из них — недоделки существующей функциональности
(хештеги вообще не сохраняются), часть — UX-недочёты новой фичи отметок. План охватывает
все проблемы + связанную проблему отображения текстовых постов.

---

## Проблемы и решения

### 1. Пустой список при открытии «Добавить пользователей» → показывать подписки

**Сейчас:** при пустом поиске шит показывает заглушку «Введите имя пользователя».
**Надо:** при пустом поиске показывать список подписок пользователя (кого он читает).

**Решение:** в `_TagUsersSheet._buildContent` при `_debouncedQuery.isEmpty` грузить
`mySubscriptionsListProvider()` (готовый провайдер, `Future<List<UserModel>>`,
`lib/core/providers/api/subscriptions_provider.dart:10`). Показывать уже выбранных +
подписки. При наличии тестовых данных список может быть пуст — это нормально.

Файл: `lib/features/feed/screens/create_post_screen.dart` (класс `_TagUsersSheetState`).

### 2. Нет очевидной кнопки «Добавить» → отметки не попадают в пост

**Сейчас:** кнопка «Готово (N)» в правом углу шапки шита — теряется на мобильном,
неочевидна.
**Надо:** явная широкая кнопка внизу шита.

**Решение:** добавить в `_TagUsersSheet` нижнюю закреплённую кнопку
`FilledButton` во всю ширину «Добавить (N)» (в `bottomNavigationBar`-стиле —
в Column после Expanded). Кнопку в шапке можно оставить или убрать.

Файл: `create_post_screen.dart`, `_TagUsersSheet.build`.

### 3. Шаг 3 (review) не показывает отметки (и хештеги видны только из _tags)

**Сейчас:** `_buildReviewStep` (строка ~601) показывает media, content, `_tags`.
Не показывает `_taggedUsers`.
**Надо:** на review-шаге показать блок «Отмечено: имя1, имя2...» и хештеги (после фикса 5).

**Решение:** добавить в `_buildReviewStep` секцию с чипами `_taggedUsers` (как на шаге 2).

Файл: `create_post_screen.dart`, `_buildReviewStep`.

### 4. Хештеги нигде не сохраняются (сквозная недоделка)

**Сейчас:**
- Frontend собирает `_tags`, но НЕ отправляет (`_submitPost`: «tags убраны»).
- `CreatePostRequest` не имеет `tags`.
- Backend: нет поля `tags` в `CreatePostDto`, `Post` entity, `PostResponseDto`.
- `PostModel.tags` всегда `[]` (backend не отдаёт).

**Надо:** реализовать хештеги end-to-end (по образцу `category_ids`/`tagged_user_ids`).

**Backend:**
- Миграция: колонка `tags text[]` в `posts` + GIN-индекс.
- `Post` entity: поле `tags: string[]`.
- `CreatePostDto`: `tags?: string[]` (`@IsString({each:true})`, MaxLength на элемент).
- `posts.service.create`: `tags` попадёт автоматически через `manager.create(Post, {...dto})`
  (как `category_ids`). Нормализация: trim, убрать `#`, дедуп, лимит (напр. 30 тегов).
- `PostResponseDto` + `posts.mapper`: поле `tags: string[]`.

**Frontend:**
- `CreatePostRequest`: добавить `tags` (`@JsonKey(name:'tags')`).
- `_submitPost`: передавать `tags: _tags` (+ убрать хештеги, введённые но не добавленные —
  см. фикс авто-добавления ниже).
- `PostModel.tags` уже есть и парсится — заработает автоматически.

**UX-фикс авто-добавления:** сейчас хештег попадает в `_tags` только по «+»/Enter.
Добавить: в `_submitPost` перед отправкой до-добавлять `_tagController.text`, если непусто
(чтобы не терять введённый, но не подтверждённый тег).

### 5. В посте не видны хештеги/отметки (+ текстовые посты)

**Причина двойная:**
- Хештеги пусты (фикс 4 решает).
- Отметки я добавил только в оверлей `post_detail_screen`, но для **текстовых постов
  без медиа** экран показывает «Нет медиа для отображения», и весь оверлей с
  content/tags/taggedUsers виден плохо/не виден.

**Надо:** корректно отображать текстовые посты (без медиа) — показывать content, хештеги,
отметки в читаемом виде, а не «Нет медиа».

**Решение:** в `post_detail_screen.dart` для постов без медиа заменить заглушку «Нет медиа»
на нормальный layout: крупный текст поста по центру/сверху на нейтральном фоне; хештеги и
отмеченные пользователи под ним (тёмным текстом, не белым оверлеем). Либо: если медиа нет —
рендерить контент как обычную карточку (не fullscreen-оверлей).

Файл: `lib/features/feed/screens/post_detail_screen.dart` (строки ~382-415 media area,
~474-666 overlay).

## Порядок реализации

1. **Backend хештеги** (миграция + entity + DTO + mapper + нормализация) — фундамент.
2. **Frontend модель** (`CreatePostRequest.tags`) + кодоген.
3. **Frontend экран создания:**
   - шит: подписки по умолчанию (1) + нижняя кнопка «Добавить» (2);
   - review-шаг: отметки + хештеги (3);
   - `_submitPost`: отправка `tags` + авто-добавление незакоммиченного тега (4).
4. **Frontend отображение** текстовых постов + хештеги/отметки (5).
5. Кодоген, `tsc --noEmit`, `flutter analyze`, сборка, ручной тест в Chrome.

## Статус реализации (по решениям пользователя)

Решения: хештеги — **убрать UI** (не сохранялись нигде); текстовые посты — **не трогать**.

Сделано:
- ✅ **Убран UI хештегов** — секция на шаге контента и review, поля `_tags`/`_tagController`,
  методы `_addTag`/`_removeTag` удалены (`create_post_screen.dart`).
- ✅ **Подписки по умолчанию** — при пустом поиске шит показывает
  `mySubscriptionsListProvider()` («Мои подписки»), выбранные — сверху; при ошибке
  загрузки не блокирует поиск.
- ✅ **Нижняя кнопка «Добавить (N)»** — закреплённая широкая кнопка внизу шита
  (вместо неочевидной «Готово» в углу).
- ✅ **Review-шаг** — показывает отмеченных пользователей чипами с аватарами.
- `flutter analyze` чист, web собран, для теста созданы 3 подписки у client@test.com.

Не удалось проверить кликом: bottom-sheet за шагом «медиа», который в Flutter web требует
файл через нативный picker — недоступен headless-плагину. Требует ручной проверки.

## Verification

- Backend: `POST /posts` с `tags` → в ответе `tags` и `taggedUsers`; GET поста тоже.
- Frontend: создать пост (с фото) → отметить людей (кнопка «Добавить», подписки видны) →
  хештеги → шаг 3 показывает и то, и другое → опубликовать → на детали поста видны
  хештеги и кликабельные отметки; текстовый пост без фото читается нормально.
